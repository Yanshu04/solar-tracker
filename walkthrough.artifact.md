## Identified Issues & Fixes

### 1. Hardcoded Site ID for Simulations
- **Issue**: In `SolarRepository.kt`, the logic for simulating storm conditions was hardcoded to check for site ID `"3"`. However, the seeded site IDs use a different format (e.g., `"global_8"`).
- **Fix**: Updated the simulation logic to use `"global_3"` (Sahara Desert Plant), matching the seeded data. This ensures that the storm simulation and related alerts trigger as intended during testing.

### 2. Misleading Solar GHI Fallback Values
- **Issue**: When the weather API (Tomorrow.io) returned `null` for Solar GHI (Global Horizontal Irradiance), the app defaulted to a hardcoded `400.0 W/m²`. This was misleading, especially during night hours or in regions with no sunlight.
- **Fix**: Changed the fallback value to `0.0 W/m²` in the repository. Additionally, updated the `PredictionScreen` to display `"No Sunlight"` when GHI is 0.0, providing a more intuitive user experience.

### 3. Missing Manifest Configuration for Android 13+
- **Issue**: Logcat reported a warning: `OnBackInvokedCallback is not enabled for the application. Set 'android:enableOnBackInvokedCallback="true"' in the application manifest.` This prevents the app from supporting modern predictive back gestures.
- **Fix**: Added `android:enableOnBackInvokedCallback="true"` to the `<application>` tag in `AndroidManifest.xml`.

### 4. Robust Date Filtering for Forecasts
- **Issue**: the logic for filtering "tomorrow's" forecast was based on simple string prefix matching, which could be unreliable across different timezones or near midnight transitions.
- **Fix**: Reimplemented the filtering logic in `SolarRepository.kt` using `Calendar` objects and explicit timezones (`site.timezone`). This ensures that the "Tomorrow's Panel Plan" screen always shows the correct 24-hour window for the selected site's local day.
